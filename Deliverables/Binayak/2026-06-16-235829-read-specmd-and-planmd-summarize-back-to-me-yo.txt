  ---
  VIPER — Project Summary

  What it is

  VIPER (Verifiable Iterative Planning with Ensemble Reasoning) is a visual AI
  planning system for physical tasks (robotic manipulation, navigation). A user
  uploads a scene image and types a goal (e.g. "move the red block to the
  bowl"), and VIPER produces an annotated, step-by-step action plan — validated
  by two independent language models.

  It fuses two published techniques:
  - PIVOT (arXiv 2402.07872): iterative visual prompting — overlay numbered
  candidate actions on the image, ask a VLM to pick the best, refine over
  multiple iterations.
  - VLMPC (RSS 2024): vision-language model predictive control — expand
  candidates into multi-step sequences, simulate outcomes with a forward model,
  score with a hierarchical cost (pixel + semantic + obstacle).

  Three-layer architecture

  ┌──────────┬────────────────────────────────────────────────┬──────────────┐
  │  Layer   │                  What it does                  │    Output    │
  ├──────────┼────────────────────────────────────────────────┼──────────────┤
  │ GENERATE │ 7-stage PIVOT+VLMPC pipeline with a single VLM │ ActionPlan   │
  ├──────────┼────────────────────────────────────────────────┼──────────────┤
  │ DEBATE   │ Claude and Qwen critique the plan in a relay,  │ DebateResult │
  │          │ converge on a verdict                          │              │
  ├──────────┼────────────────────────────────────────────────┼──────────────┤
  │ OUTPUT   │ Combined plan + verdict + transcript           │ ViperResult  │
  └──────────┴────────────────────────────────────────────────┴──────────────┘

  The layers are strictly separable: GENERATE works and is testable
  independently; DEBATE only ever sees a packaged DebateArtifact, never pipeline
  internals.

  Key architectural constraints

  - Every VLM call goes through VLMInterface in src/vlm/interface.py — swapping
  mock → real is a config change, not a code edit.
  - All image drawing goes through src/visualization/overlay.py exclusively.
  - All cross-module data is typed dataclasses from src/models.py — no bare
  dicts.
  - All tunables live in config/pipeline.yaml — no hard-coded numbers in src/.
  - The debate models never call each other directly; the orchestrator relays
  text between them.

● Confirmed. Here is what I understand about the architecture:

  Three independent layers, strict separation. GENERATE (7-stage pipeline) →
  DEBATE (relay) → ViperResult. GENERATE must be fully functional and testable
  before DEBATE is touched. DEBATE only consumes a DebateArtifact — it never
  reaches back into pipeline internals or sees ground truth.

  Single VLM entry point. Every VLM call in GENERATE goes through VLMInterface
  (the ABC in src/vlm/interface.py). No stage talks to an SDK directly.
  Switching vlm_backend: mock → anthropic in config/pipeline.yaml is the entire
  swap — zero code changes. The debate models (AnthropicVLM, QwenVLM)
  additionally implement critique_plan, which is separate from the four pipeline
  methods.

  Typed dataclasses only across module boundaries. src/models.py is the single
  source of all data shapes for both layers. No bare dicts passed between
  modules.

  Drawing is monopolized by overlay.py. No pipeline stage or debate module draws
  on images. Only src/visualization/overlay.py does, and every drawing function
  returns a new PIL Image without mutating its input.

  All tunables in config/pipeline.yaml. Hard-coding any numeric parameter in
  src/ is a bug.

  MockVLM is deterministic. numpy.random.default_rng(seed) for all randomness.
  Tests are repeatable and free.

  Debate relay semantics. Claude opens, Qwen responds seeing Claude's turn,
  Claude responds seeing Qwen's, alternating up to max_rounds. Convergence =
  latest verdicts match. Tie-break = most conservative verdict (reject > amend >
  endorse). The orchestrator is the sole caller — models only ever see each
  other's relayed text.

  Gradio UI contains zero logic. It calls run_pipeline / run_viper and maps
  results to display components. Nothing else.

  # VIPER Phase Workflow

  ## The loop (repeat for every phase, in order)

  1. **Read the gate first.** Before touching any code, state the current phase
     number, its goal sentence, and its exact gate command/criterion from the
     table below. Confirm the previous phase's gate already passed.

  2. **Build the simplest thing that passes the gate.** Do not add features,
     abstractions, or polish beyond what the gate requires. Items marked
     `[OPTIONAL]` in plan.md are skipped until the entire system works
     end-to-end.

  3. **Run the gate command.** After implementation, execute the gate command
     exactly as listed. Do not move on until it passes.

  4. **Report status, then stop.** After the gate passes, output a short
     status report (phase number, gate command used, pass/fail, any
     notable decisions). Do not begin the next phase without explicit
     user approval.

  5. **Never start a phase until the previous phase's gate has passed.**
